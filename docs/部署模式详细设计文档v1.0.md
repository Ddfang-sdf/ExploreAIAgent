## 1. 总体设计

### 1.1 模块定位

部署模式模块是系统的**对外服务层**。它负责解析命令行参数、加载配置、初始化核心模块，然后按选定的模式（CLI 或 Web）启动系统，将用户输入路由到 Orchestrator 并返回结果。

**核心职责**：

1. 命令行参数解析与验证
2. 系统配置加载（`AppConfig::load`）
3. 核心模块组装（Orchestrator、各 Agent、上下文工具）
4. 按模式启动用户交互通道（CLI REPL 或 HTTP Server）
5. 会话生命周期管理（ECT 的创建、复用、销毁）

### 1.2 核心原则

| 原则 | 说明 |
|:---|:---|
| **薄壳入口** | CLI 和 Web 仅负责 IO 和会话管理，所有 AI 逻辑通过 `Orchestrator::run()` 调用 |
| **ECT 会话隔离** | 每个 session 持有独立的 `ExplorationContextTool` 实例，由 CLI/Web 层自行管理 |
| **配置驱动** | 系统行为完全由配置文件控制，入口层不做硬编码决策 |
| **统一启动流程** | CLI 和 Web 共享模块初始化过程，仅最终交互通道不同 |

### 1.3 启动流程

```mermaid
flowchart TD
    A[进程启动] --> B[解析命令行参数]
    B --> C[加载 AppConfig]
    C --> D[初始化 WorkDirInitializer]
    D --> E[check_workspace]
    E --> F[初始化 ApiAdapter]
    F --> G[初始化 ToolRegistry]
    G --> H[初始化 ConversationManager]
    H --> I[初始化 Orchestrator]
    I --> J{部署模式?}

    J -- CLI --> K[创建 ExplorationContextTool]
    K --> L[进入 REPL 循环]
    L --> M{/exit?}
    M -- 否 --> N[读取 stdin]
    N --> O[Orchestrator::run]
    O --> P[输出 stdout]
    P --> L
    M -- 是 --> Q[退出]

    J -- Web --> R[创建 SessionManager]
    R --> S[启动 HTTP Server]
    S --> T[监听端口]
    T --> U[接受连接]
    U --> V[/chat 请求]
    V --> W[获取或创建 ECT]
    W --> X[Orchestrator::run]
    X --> Y[返回 JSON 响应]
    Y --> T
```

## 2. CLI 模式详细设计

### 2.1 启动入口

```bash
explore-agent --config config.yaml --workspace ./nanahash
```

**命令行参数**：

| 参数 | 必选 | 默认值 | 说明 |
|:---|:---|:---|:---|
| `--config` | 否 | `./config.yaml` | 配置文件路径 |
| `--workspace` | 否 | 配置文件中的 `workspace.path` | 工作目录（覆盖配置文件） |
| `--mode` | 否 | `cli` | 部署模式：`cli` 或 `serve` |

### 2.2 REPL 循环

```rust
pub async fn run_cli(config: AppConfig) -> Result<(), String> {
    // 组装模块（与 Web 共享）
    let (adapter, registry, cm, orchestrator) = assemble_core(&config)?;

    // CLI 模式：单个 ECT 实例，整个 REPL 复用
    let mut ect = ExplorationContextTool::new("cli-session".to_string());
    ect.configure(&config.exploration, &config.context);

    let stdin = std::io::stdin();
    let mut line = String::new();

    println!("Explore AI Agent (CLI mode). Type /exit to quit.");
    loop {
        print!(">>> ");
        std::io::Write::flush(&mut std::io::stdout()).unwrap();
        line.clear();
        stdin.read_line(&mut line).map_err(|e| e.to_string())?;
        let input = line.trim().to_string();

        if input.is_empty() { continue; }
        if input == "/exit" || input == "/quit" { break; }

        match orchestrator.run(&input, &mut ect).await {
            Ok(answer) => println!("{}\n", answer),
            Err(e) => eprintln!("[ERROR] {}\n", e),
        }

        // 对话上下文精炼检查
        cm.check_and_refine("cli-session", &input).await?;
    }
    Ok(())
}
```

**REPL 设计细节**：

| 细节 | 说明 |
|:---|:---|
| 会话标识 | 固定为 `"cli-session"`，单进程单 session |
| ECT 生命周期 | REPL 开始时创建，结束时释放；中间不做销毁重建 |
| 错误处理 | 单轮失败不影响 REPL 继续；LLM 错误打印到 stderr |
| 退出命令 | `/exit` 或 `/quit`；Ctrl+C 直接终止 |

### 2.3 数据结构

CLI 模式不需要额外的数据结构。输入为 `String`，输出为 `String`。


**命令行参数**：

| 参数 | 必选 | 默认值 | 说明 |
|:---|:---|:---|:---|
| `--config` | 否 | `./config.yaml` | 配置文件路径 |
| `--port` | 否 | `8080` | HTTP 监听端口 |

### 3.2 API 路由

| 端点 | 方法 | 说明 |
|:---|:---|:---|
| `/chat` | POST | 核心聊天接口 |
| `/health` | GET | 健康检查 |

### 3.3 Handler 实现

#### 3.3.1 共享状态

```rust
pub struct AppState {
    pub orchestrator: Orchestrator,
    pub conversation_manager: ConversationManager,
    pub sessions: Mutex<HashMap<String, ExplorationContextTool>>,
    pub config: AppConfig,
}
```

| 字段 | 说明 |
|:---|:---|
| `orchestrator` | 共享 Orchestrator 实例（无状态，所有 session 共用） |
| `conversation_manager` | 共享（内部 HashMap<session_id> 隔离），对话上下文跨轮次 |
| `sessions` | 探索上下文按 session_id 隔离，`Mutex` 保护并发写 |
| `config` | 系统配置，用于新 session 的 ECT 初始化 |

#### 3.3.2 POST /chat

**请求**：

```json
{
  "session_id": "7f3a-4b2c",  
  "question": "BooleanValidator 有哪些参数？"
}
```

`session_id` 为可选字段——首次请求可省略，服务端自动创建并返回。

**响应**：

```json
{
  "code": 0,
  "session_id": "7f3a-4b2c",
  "answer": "BooleanValidator 支持两个配置参数..."
}
```

**处理流程**：

```mermaid
flowchart TD
    A[接收 POST /chat 请求] --> B[解析 JSON body]
    B --> C{解析成功?}
    C -- 否 --> D[返回 code=1, msg='Invalid JSON']
    C -- 是 --> E{question 非空?}
    E -- 否 --> F[返回 code=2, msg='question is required']
    E -- 是 --> G{session_id 存在?}
    G -- 否 --> H[生成新 session_id<br/>uuid v4 前 8 位]
    G -- 是 --> I[使用已有 session_id]
    H --> J{sessions 中存在该 id?}
    I --> J
    J -- 否 --> K[创建新 ECT<br/>configure 从 config 读取阈值]
    K --> L[插入 sessions HashMap]
    J -- 是 --> M[从 sessions 取出 ECT]
    L --> M
    M --> N[orchestrator.run(question, &mut ect)]
    N --> O{成功?}
    O -- 是 --> P[返回 code=0, session_id, answer]
    O -- 否 --> Q[返回 code=3, msg=错误信息]
```

**架构说明**：为支持可测试性，业务逻辑抽离为纯函数 `handle_chat_request`。axum Handler 仅负责提取 `State` 和 `Json` 并转发调用，不做业务判断。测试代码直接调用 `handle_chat_request`。

**纯函数签名**：

```rust
pub async fn handle_chat_request(
    body: ChatRequest,
    state: &AppState,
) -> ChatResponse
```

**axum Handler（薄壳）**：

```rust
async fn chat_handler(
    State(state): State<Arc<AppState>>,
    Json(body): Json<ChatRequest>,
) -> Json<ChatResponse> {
    Json(handle_chat_request(body, &state).await)
}
```

**业务逻辑**：

```rust
pub async fn handle_chat_request(
    body: ChatRequest,
    state: &AppState,
) -> ChatResponse {
    // 校验
    if body.question.trim().is_empty() {
        return ChatResponse::error(2, "question is required");
    }

    // 获取或创建 session
    let session_id = body.session_id
        .unwrap_or_else(|| uuid::Uuid::new_v4().to_string()[..8].to_string());

    let mut sessions = state.sessions.lock().unwrap();
    let ect = sessions.entry(session_id.clone())
        .or_insert_with(|| {
            let mut ect = ExplorationContextTool::new(session_id.clone());
            ect.configure(&state.config.exploration, &state.config.context);
            ect
        });

    // 调用 Orchestrator
    match state.orchestrator.run(&body.question, ect).await {
        Ok(answer) => ChatResponse {
            code: 0,
            session_id,
            answer: Some(answer),
            error: None,
        },
        Err(e) => {
            let code = if e.contains("LLM") || e.contains("retry") { 3 }
                else if e.contains("context") || e.contains("ECT") { 4 }
                else { 5 };
            ChatResponse::error(code, &e)
        }
    }
}
```

#### 3.3.3 GET /health

**响应**：`{"status": "ok"}`

无业务逻辑，仅用于负载均衡器或监控探活。

### 3.4 会话管理

| 维度 | 设计 |
|:---|:---|
| ECT 创建时机 | 首次 `/chat` 请求该 `session_id` 时延迟创建 |
| ECT 生命周期 | 跟随 session，由 `SessionManager` 的 HashMap 持有 |
| 清理策略 | 可配置 TTL（默认 30 分钟），后台定时任务扫描过期 session 并移除 |
| 并发模型 | `Mutex<HashMap<...>>` 保护 session 的增删改查；同一 session 的并发调用由 axum 的异步 handler 天然串行化 |
| TTL 实现 | 每个 session 记录 `last_access: Instant`；后台每隔 5 分钟扫描一次，移除超过 TTL 的 |

### 3.5 数据结构

```rust
#[derive(Debug, Deserialize)]
pub struct ChatRequest {
    #[serde(default)]
    pub session_id: Option<String>,
    pub question: String,
}

#[derive(Debug, Serialize)]
pub struct ChatResponse {
    pub code: i32,
    pub session_id: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub answer: Option<String>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub error: Option<String>,
}

impl ChatResponse {
    pub fn error(code: i32, msg: &str) -> Self {
        ChatResponse {
            code,
            session_id: String::new(),
            answer: None,
            error: Some(msg.to_string()),
        }
    }
}
```

**错误码定义**：

| code | 含义 |
|:---|:---|
| `0` | 成功 |
| `1` | 请求体 JSON 解析失败 |
| `2` | question 为空 |
| `3` | LLM 调用或适配层错误（含重试耗尽） |
| `4` | 探索上下文或对话上下文操作失败 |
| `5` | 其他内部错误 |

## 4. 子系统命令设计

通过命令行子命令区分模式：

```bash
# CLI 模式（默认）
explore-agent --config config.yaml

# Web 模式
explore-agent serve --config config.yaml --port 8080
```

子命令由 `clap` crate 解析。`serve` 是独立子命令；无子命令时默认为 CLI 模式。

## 5. 错误处理

| 场景 | CLI 模式 | Web 模式 |
|:---|:---|:---|
| 配置文件不存在 | 打印错误到 stderr，退出 | 打印错误到 stderr，退出 |
| 配置文件格式错误 | 打印错误到 stderr，退出 | 打印错误到 stderr，退出 |
| 工作目录不存在 | 打印 WARN 日志，继续启动 | 打印 WARN 日志，继续启动 |
| LLM 调用或适配层错误 | 打印错误到 stderr，REPL 继续 | 返回 `code=3` 及错误信息 |
| 探索上下文或对话上下文操作失败 | 打印错误到 stderr，REPL 继续 | 返回 `code=4` 及错误信息 |
| 其他内部错误 | 打印错误到 stderr，REPL 继续 | 返回 `code=5` 及错误信息 |
| session_id 对应的 ECT 不存在 | — | 自动创建 |
| HTTP 请求体 JSON 非法 | — | 返回 `code=1` |
| question 为空 | 跳过本轮 | 返回 `code=2` |

## 7. 附录

### 7.1 依赖清单

| crate | 用途 | 模式 |
|:---|:---|:---|
| `clap` | 命令行参数解析 | CLI + Web |
| `axum` | HTTP 框架 | Web |
| `uuid` | 生成 session_id | Web |
| `tokio` | 异步运行时（已有） | CLI + Web |

### 7.2 模块结构

```
src/
  main.rs           # 入口：参数解析 → 分支到 CLI 或 Web
  web/
    mod.rs          # Web 模式：路由 + handler
    session.rs      # SessionManager：HashMap + TTL
```

### 7.3 不变式与约束

| 约束 | 说明 |
|:---|:---|
| **模块初始化共享** | CLI 和 Web 通过 `assemble_core()` 共享初始化逻辑 |
| **ECT 不跨 session** | 每个 session_id 持有独立的 ECT 实例 |
| **Orchestrator 无状态** | 同一 Orchestrator 实例服务所有 session |
| **Web 无状态路由** | 每次 `/chat` 请求独立处理，状态仅由 session_id 关联 |
